# Resources and localization

JustHostMC uses the Windows App SDK resource system (MRT Core). Keep static UI
copy in XAML resources, and use the C# localization boundary only when runtime
state, runtime values, a backend-provided key, or an unavoidable imperative UI
owner determines the text.

Microsoft's platform references are:

- [Localize strings in your UI and app package manifest](https://learn.microsoft.com/windows/apps/windows-app-sdk/mrtcore/localize-strings)
- [ResourceDictionary and XAML resource references](https://learn.microsoft.com/windows/apps/develop/platform/xaml/xaml-resource-dictionary)
- [Make your app localizable](https://learn.microsoft.com/windows/apps/design/globalizing/prepare-your-app-for-localization)

## Directory layout

The app's resource roots are:

```text
app/JustHostMC.App/
  Strings/
    en-US/Resources.resw
    zh-TW/Resources.resw
  Assets/
```

`en-US` is the base and fallback language. It is set by `<DefaultLanguage>` in
`JustHostMC.App.csproj` and listed first in `Package.appxmanifest`. The manifest
currently declares `en-US` and `zh-TW`; both locale files must expose identical
resource-name sets.

## Choosing XAML or C# ownership

| Text source | Owner |
|---|---|
| Fixed label, heading, button text, dialog chrome, placeholder, tooltip, or accessibility name | XAML element with `x:Uid` |
| Runtime state selects one of several messages | `ILocalizer.Get` |
| Engine/provider supplies a `LocalizedMessage` key | `ILocalizer.Get` |
| Message contains runtime values | `ILocalizer.Get` with placeholders |
| Static text belongs to an imperative control that cannot safely be represented in XAML | Documented imperative exception |
| Diagnostic detail that is not user-facing | Do not localize |

The current imperative exception is the mod shop's dynamically created
`SelectorBarItem` tooltip. Replacing that owner with a plain managed
`ItemsSource` and an `x:Bind` template triggers the project's documented WinUI
startup crash. Record any new exception in
[`resource-lookup-inventory.md`](resource-lookup-inventory.md); do not treat
imperative lookup as the default.

## Static strings: XAML first

Give the owning element an `x:Uid` and name the `.resw` entry with the exact
native property:

```xml
<Button x:Uid="CommonSaveButton" />

<data name="CommonSaveButton.Content" xml:space="preserve">
  <value>Save</value>
</data>
```

Do not also set `Content="Save"` in XAML or assign it from C#. One UID must not
be shared by different control types when their target properties differ; for
example, a `Button` uses `.Content` while a `TextBlock` uses `.Text`.

Attached properties use the full property identifier. A tooltip is written as:

```xml
<Button x:Uid="RefreshButton" />

<data name="RefreshButton.ToolTipService.ToolTip" xml:space="preserve">
  <value>Refresh</value>
</data>
```

The same rule applies to accessibility properties. This repository uses keys
such as
`CommonCopyAddressButton.[using:Microsoft.UI.Xaml.Automation]AutomationProperties.Name`.
The policy tests verify that every `x:Uid` resolves and that its properties are
compatible with the element type.

## Dynamic strings: C# exception

Application and backend code use dotted semantic identifiers. Only
`LocalizationService` converts dots to MRT's forward-slash path at the
`ResourceLoader` boundary. For the illustrative resource `Menu.File.Open`:

```csharp
var label = _localizer.Get("Menu.File.Open");
```

The internal MRT call is equivalent to:

```csharp
resourceLoader.GetString("Menu/File/Open");
```

Never normalize the key to `Menu_File_Open`, and do not pass `Menu/File/Open`
from normal application code. Backend dotted keys remain unchanged until this
boundary. Missing-resource fallback is for runtime safety, not permission to
skip the paired resource entries or policy tests.

## Common copy and duplicate policy

Reuse copy only when its semantics and target property are compatible. Prefer a
`Common*` UID for interchangeable actions such as Save, Close, Remove, Install,
Refresh, Copy, or Next. Coincidentally identical English words remain separate
when translator context differs. Native `x:Uid` also requires separate entries
when owners target incompatible properties such as `.Content`, `.Text`,
`.PrimaryButtonText`, or `.CloseButtonText`.

Every unavoidable duplicate must be approved in `DuplicateResourceRules` in
`ResourcePolicyTests.cs`, and every member in both locales must carry the exact
structured comment generated by that rule. The accepted categories are
`DUPLICATE PROPERTY-CONTEXT:`, `DUPLICATE SEMANTIC-HOMOGRAPH:`, and
`DUPLICATE RUNTIME-CONTRACT:`. For example:

```xml
<comment>DUPLICATE PROPERTY-CONTEXT: canonical Common.Save; WinUI x:Uid targets a different property.</comment>
```

The design shorthand `INTENTIONAL DUPLICATE:` is not an allowlist by itself;
the policy test deliberately rejects a generic comment so an unrelated new
duplicate cannot silently inherit an old exception.

## Placeholders and translator comments

Use named placeholders for new `ILocalizer` messages:

```xml
<data name="Server.MemoryValue" xml:space="preserve">
  <value>{memory} MB</value>
  <comment>Placeholders: {memory} = memory in megabytes.</comment>
</data>
```

```csharp
_localizer.Get("Server.MemoryValue", ("memory", memoryMb.ToString()));
```

Each locale must contain the same placeholder occurrences, including repeated
occurrences, not merely the same distinct set. Each formatted resource also
needs a translator comment that names and explains every placeholder.

`PlayerDialogBase.TitleFormat` and `Shop.ResultSummary` are retained
`string.Format` call sites and therefore use documented positional placeholders
such as `{0}` and `{1}`. Treat these as existing exceptions; use named
placeholders with `ILocalizer` for new or changed runtime formats.

## Images and file resources

Store packaged images and other file resources under `Assets/`. Reference a
package-relative file with an absolute package URI such as
`ms-appx:///Assets/Square44x44Logo.png`; manifest paths use their manifest form,
for example `Assets\StoreLogo.png`.

Provide MRT scale and language-qualified variants when the same logical asset
needs DPI- or locale-specific files, following the Windows qualifier naming and
folder conventions. Keep the logical URI stable and let MRT select the best
candidate. Ensure every required variant is included in the project/package and
does not collide with another qualified candidate.

Do not embed localizable words in images. Put meaningful text in `.resw` and
expose accessible text outside the bitmap with a visible label or
`AutomationProperties.Name`. Decorative images should not receive misleading
accessible names.

## Manifest resources

`Package.appxmanifest` references `ms-resource:AppDisplayName` and
`ms-resource:AppDescription`; those simple identifiers must exist in every
locale. It declares these supported languages:

```xml
<Resources>
  <Resource Language="en-US" />
  <Resource Language="zh-TW" />
</Resources>
```

`JustHostMC.App.csproj` sets `<DefaultLanguage>en-US</DefaultLanguage>` and keeps
`Language=en-US;zh-TW` in `AppxDefaultResourceQualifiers` so packaged builds
retain both languages. Keep the project default, manifest language list, package
qualifiers, and locale directories synchronized. When a requested language is
unavailable, MRT falls back to the `en-US` resources.

## Adding a language or resource

For a new or changed resource:

1. Choose XAML or dynamic ownership with the table above.
2. Add the exact same key to both `en-US` and `zh-TW`. Add the matching `x:Uid`
   or dotted C# lookup, not both.
3. Add translator comments for every placeholder and every approved duplicate.
4. Run the focused policy tests and the x64 app build below.

For a new language:

1. Create `Strings/<BCP-47-tag>/Resources.resw` by copying the complete `en-US`
   key set, then translate values without changing keys or placeholder
   occurrences.
2. Add `<Resource Language="<BCP-47-tag>" />` to `Package.appxmanifest` for
   packaged builds.
3. Add the tag to `AppxDefaultResourceQualifiers` so the Store/MSIX build keeps
   it, and leave `DefaultLanguage` unchanged unless the product fallback policy
   is intentionally changing.
4. Extend any locale enumeration in `ResourcePolicyTests`, then run the tests,
   package build, and both-language UI smoke tests.

## Validation and troubleshooting

Run the resource policy suite:

```powershell
rtk dotnet test app/JustHostMC.Core.Tests/JustHostMC.Core.Tests.csproj --filter FullyQualifiedName~ResourcePolicyTests -p:SkipEngineBuild=true -p:SkipProtobufGenerate=true
```

Build the WinUI app for an explicit supported platform:

```powershell
rtk dotnet build app/JustHostMC.App/JustHostMC.App.csproj -p:Platform=x64 -p:SkipEngineBuild=true -p:SkipProtobufGenerate=true
```

Also smoke-test every navigable page and dialog in `en-US` and `zh-TW` after
changing XAML UIDs, attached properties, manifest resources, or qualifiers. A
successful build does not catch every runtime `XamlParseException`.

If a resource is missing at runtime:

- Confirm the `.resw` key exists in both locales and that its `x:Uid` property
  matches the element type.
- Confirm C# supplies a dotted key and `LocalizationService` alone converts it
  to a forward-slash MRT path.
- Confirm the BCP-47 folder, manifest language, `DefaultLanguage`, and package
  qualifiers agree.
- Rebuild the package so its PRI index is regenerated; do not edit a generated
  `.pri` file.
- When deeper inspection is necessary, run `makepri.exe dump /if
  <path-to-resources.pri> /of <path-to-dump.xml>` from a Windows/Visual Studio
  developer shell and inspect the resulting XML for the resource and
  qualifiers. In Git Bash, set `MSYS_NO_PATHCONV=1` so `/if` and `/of` are not
  rewritten as Unix paths.

Finally run `rtk git diff --check` before committing resource or documentation
changes.
